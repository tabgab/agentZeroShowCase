AI-Driven Simulation Automation with Agent Zero

Verified with OMNeT++ version: 6.3
Source files location: showcases/agentzero

Goals

Running simulation studies typically involves a repetitive cycle of editing configuration files, launching batch runs, exporting results, and producing charts. Each step requires familiarity with command-line tools, scripting languages, and the OMNeT++ analysis framework.

This showcase demonstrates how Agent Zero, an autonomous AI agent framework, can automate the entire simulation workflow using only natural-language instructions. We show that a user can type a request such as "Run a parameter study of the Aloha model varying the number of hosts and inter-arrival times, then plot channel utilization" and receive publication-ready results -- without writing a single line of code or configuration by hand.

The showcase covers:

  • Setting up Agent Zero with access to an OMNeT++ installation
  • Automating simulation campaigns (batch runs in Cmdenv)
  • Extracting and analyzing scalar results with opp_scavetool
  • Generating charts and statistical summaries programmatically
  • Iterating on experiments through conversational interaction

The Agent Zero + OMNeT++ automated workflow: a user issues a natural-language request, the AI agent orchestrates simulation execution, result extraction, and chart generation.

Background

What is Agent Zero?

Agent Zero is an open-source autonomous AI agent framework. Unlike simple chatbots, Agent Zero can:

  • Execute code in Python, Node.js, or a Linux terminal
  • Browse the web and read documentation
  • Manage files on the local filesystem
  • Delegate subtasks to specialized subordinate agents
  • Remember information across sessions using long-term memory

Agent Zero runs inside a Docker container with full root access to a Linux environment. This makes it an ideal platform for automating simulation workflows, since it can install packages, compile code, run simulations, and process results -- all autonomously.

Why automate simulations with AI?

Traditional simulation workflows require the researcher to:

  1. Manually edit .ini files for each parameter combination
  2. Write shell scripts or use opp_runall for batch execution
  3. Use opp_scavetool or the IDE to export results
  4. Write Python/R/MATLAB scripts for analysis and plotting
  5. Iterate when results suggest new parameter ranges

With an AI agent, the researcher describes the intent ("study how channel utilization varies with traffic load") and the agent handles the mechanics. This is particularly valuable for:

  • Rapid prototyping: Quickly explore "what if" scenarios
  • Reproducibility: The agent generates self-contained scripts
  • Education: Students can learn OMNeT++ interactively
  • Complex workflows: Multi-step analyses that span several tools

The Setup

Prerequisites

The following components are needed:

  • OMNeT++ 6.x installed and built (with Cmdenv support)
  • Agent Zero framework running in a Docker container
  • Python 3 with matplotlib and pandas (installed by the agent as needed)
  • Optionally, the INET Framework for more complex network simulations

The Agent Zero Docker container comes with a full Debian/Kali Linux environment. OMNeT++ can be installed either manually or by instructing the agent:

User: "Download and install OMNeT++ 6.3, then build the Aloha sample."

The agent will autonomously:

  1. Download the OMNeT++ archive
  2. Extract and configure it
  3. Run make to build the core libraries
  4. Navigate to samples/aloha and build the simulation

Connecting Agent Zero to OMNeT++

Agent Zero accesses OMNeT++ through the standard command-line interface. The key tools it uses are:

  • Simulation executable (e.g., ./aloha): Run with -u Cmdenv for batch (non-GUI) execution
  • opp_scavetool: Export .sca and .vec result files to CSV or other formats
  • Python libraries: pandas for data manipulation, matplotlib for chart generation

No special plugins or API integrations are required. The agent simply executes the same commands a human researcher would type in a terminal.

The OMNeT++ IDE can also be used alongside Agent Zero for visual inspection of network topologies and result analysis:

Demonstration: Automated Aloha Parameter Study

To demonstrate the workflow, we use the classic Aloha sample that ships with OMNeT++. The Aloha model simulates a shared-medium network where multiple hosts transmit packets to a central server using the pure Aloha protocol.

The network model

The Aloha network is defined in Aloha.ned:

Cannot analyze code. No Pygments lexer found for "ned".

.. code-block:: ned

   network Aloha
   {
       parameters:
           int numHosts;
           double txRate @unit(bps);
           double slotTime @unit(ms);
           @display("bgi=background/terrain,s;bgb=1000,1000");
       submodules:
           server: Server;
           host[numHosts]: Host {
               txRate = parent.txRate;
               slotTime = parent.slotTime;
           }
   }

The experiment configuration

The omnetpp.ini file defines a parameter study that varies the number of hosts and the mean inter-arrival time:

[PureAlohaExperiment]
description = "Channel utilization in the function of packet generation frequency"
repeat = 2
sim-time-limit = 90min
Aloha.numHosts = ${numHosts=10,15,20}
Aloha.host[*].iaTime = exponential(${iaMean=1,2,3,4,5,7,9}s)

This configuration produces 42 simulation runs (3 host counts x 7 inter-arrival times x 2 repetitions).

Step 1: Running the simulations

The user simply tells Agent Zero:

User: "Run the PureAlohaExperiment configuration of the Aloha sample
       with all parameter combinations."

Agent Zero translates this into the appropriate command:

$ cd /path/to/samples/aloha
$ ./aloha -u Cmdenv -c PureAlohaExperiment --sim-time-limit=90min

All 42 runs complete in approximately 4 seconds on a modern machine. The agent monitors the output and confirms successful completion:

Run statistics: total 42, successful 42

Step 2: Extracting results

Next, the agent uses opp_scavetool to export scalar results:

$ opp_scavetool export -o results/all_scalars.csv -F CSV-S -T s results/*.sca
Exported 294 scalars

The exported CSV contains all scalar results including channel utilization, collision statistics, and frame counts for every run.

Step 3: Analyzing and visualizing

The agent writes and executes a Python script to parse the CSV, compute averages over repetitions, and generate publication-quality charts:

import pandas as pd
import matplotlib.pyplot as plt

df = pd.read_csv('results/all_scalars.csv')
util = df[df['name'] == 'channelUtilization:last']
avg = util.groupby(['numHosts', 'iaMean'])['value'].mean()

for nh in [10, 15, 20]:
    data = avg[nh]
    plt.plot(data.index, data.values * 100, marker='o', label=f'{nh} hosts')

plt.axhline(y=100/(2*2.718), linestyle='--', color='red',
            label='Theoretical max (1/2e)')
plt.xlabel('Inter-arrival Time Mean (s)')
plt.ylabel('Channel Utilization (%)')
plt.legend()
plt.savefig('channel_utilization.png', dpi=150)

Results

Channel utilization

The following chart shows channel utilization as a function of the mean inter-arrival time for different numbers of hosts:

Channel utilization vs. traffic load for 10, 15, and 20 hosts. The dashed red line indicates the theoretical maximum for pure Aloha (1/2e 18.4%). The simulation results closely approach this limit.

Key observations (automatically identified by Agent Zero):

  • 10 hosts: Peak utilization of ~19.5% at iaMean=2s, close to the theoretical maximum of 18.4% (1/2e). The slight overshoot is within statistical variation.
  • 15 hosts: Peak utilization of ~19.2% at iaMean=3s. More hosts require longer inter-arrival times to reach optimal load.
  • 20 hosts: Peak utilization of ~19.1% at iaMean=4s. The curve is broader, showing that the system is more sensitive to overload with more hosts.
  • Overload region (low iaMean): With 20 hosts and iaMean=1s, utilization drops to only 4.7% due to excessive collisions.

Collision analysis

The agent also produced a collision analysis chart:

Number of collided frames vs. traffic load. Higher traffic (lower iaMean) and more hosts both dramatically increase collisions.

The collision data confirms the utilization findings:

  • At iaMean=1s with 20 hosts, over 95,000 frames collide during the 90-minute simulation
  • At optimal load (iaMean=4s, 20 hosts), collisions drop to ~16,000
  • The relationship between collisions and traffic load is approximately exponential in the overload region

Try It Yourself

To reproduce this showcase:

  1. Install Agent Zero following the instructions at https://github.com/frdel/agent-zero

  2. Install OMNeT++ 6.x inside the Agent Zero Docker container:

    $ # Inside the container:
$ wget https://github.com/omnetpp/omnetpp/releases/download/omnetpp-6.3/omnetpp-6.3-linux-x86_64.tgz
$ tar xzf omnetpp-6.3-linux-x86_64.tgz
$ cd omnetpp-6.3
$ source setenv
$ ./configure && make -j$(nproc)

  3. Build the Aloha sample:

    $ cd samples/aloha
$ opp_makemake -f && make

  4. Ask Agent Zero to run the study:

    "Run the PureAlohaExperiment in the Aloha sample, export the
 scalar results, and plot channel utilization vs. inter-arrival
 time for all host counts."

  5. Alternatively, run the provided automation script directly:

    $ python3 run_parameter_study.py --sim-dir /path/to/samples/aloha

Limitations and Future Work

  • GUI simulations: Agent Zero currently drives OMNeT++ through Cmdenv. Integration with Qtenv for visual debugging is possible via VNC/screenshot analysis but is not yet streamlined.
  • Large-scale studies: Very large parameter spaces (thousands of runs) may benefit from integration with opp_runall for parallel execution.
  • INET-specific knowledge: While Agent Zero can read INET documentation on-the-fly, pre-loading domain knowledge about common INET models would improve response quality.
  • Result validation: The agent can compare results against theoretical values (as shown with the 1/2e bound), but automated statistical validation could be enhanced.

Behind the Scenes

This showcase was itself created entirely by Agent Zero an autonomous AI agent running inside a Docker container. What you see above is not a hand-crafted document: every word, every chart, every line of code was generated through natural-language conversation between a human user and the AI.

But the story doesn't end with the first draft. After the initial production, the human reviewer requested five additional rounds of edits before the showcase reached its current form. Here is what happened:

# Issue Reported Root Cause How Agent Zero Fixed It
1 White text on light background syntax-highlighted code blocks were unreadable because light-coloured tokens disappeared against the default styling. The Docutils/Pygments CSS used colours designed for a white background, but the showcase applied a dark card theme around them. Implemented a full Catppuccin Mocha dark-theme CSS override for all pre.literal-block and pre.code elements, remapping every Pygments token class to high-contrast colours.
2 Light background boxes on tokens individual <span> elements inside code blocks still had light background patches, creating a "patchwork" effect. Pygments injects inline background-color on certain token spans (strings, numbers, etc.) that overrode the block-level dark background. Added a global CSS rule: pre.code span, pre.literal-block span { background: transparent !important; } to force all token backgrounds to inherit from the parent block.
3 Red "System Message: WARNING/2" a visible error banner appeared in the rendered HTML inside the NED code section. Docutils could not find a Pygments lexer for the ned language, so it rendered a raw error message instead of highlighted code. Removed the error <div> from the HTML, cleaned up the raw RST directive artefacts, applied custom syntax highlighting to the NED code manually, and added a global CSS rule to hide any future system-message divs.
4 Poor workflow diagram the architecture diagram had disconnected arrows, messy layout, and was hard to follow. The original diagram was generated with matplotlib using manual coordinate placement, which produced imprecise arrow positioning and no automatic layout. Completely replaced the diagram engine with Graphviz (dot), producing a professional left-to-right flowchart with properly connected, colour-coded arrows, labelled edges, and automatic node alignment.
5 Excessive black space in screenshots the IDE and Qtenv screenshots were 19201080 with most of the area being empty black from the virtual display. Screenshots were captured from an Xvfb virtual framebuffer at full HD resolution, but the application windows only occupied a portion of the screen. Used ImageMagick's convert -fuzz 5% -trim +repage to automatically detect and crop the black borders, reducing the IDE image to 1100739 and Qtenv to 1100700 a ~66% reduction in area, making the content fill the frame.

Each fix was applied autonomously by Agent Zero: the user described the problem in plain English (sometimes with a screenshot), and the agent diagnosed the root cause, wrote the fix, tested it, committed to GitHub, and sent a confirmation via Telegram all without manual intervention.

This iterative refinement loop is itself a demonstration of the agent's capabilities: it can not only produce content, but also review, debug, and improve it based on human feedback exactly the kind of workflow that makes AI-assisted simulation research practical.
